# 劳动者港湾智能助手开发规划

更新时间：2026-07-08

## 1. 当前项目状态

当前仓库已经连接 GitHub，主分支为 `main`。仓库内已有：

- `劳动者港湾智能助手_PRD_V1.0.md`：核心产品需求文档。
- `connect-github.ps1`：本地 GitHub 连接与首次上传脚本。
- `docs/开发规划.md`：本规划文档。

当前尚未建立应用代码、数据库结构、接口服务、前端页面或自动化测试。因此下一步不是直接堆功能，而是先完成一个可运行、可演示、可持续迭代的项目骨架。

## 2. 规划依据

本规划主要依据 PRD 中的 MVP 收敛原则、功能优先级、核心业务流程、数据表设计、页面需求和验收标准。

用户提供的调研 PDF 为 16 页纯图版，当前未检测到可直接抽取的文字层。短期内将其作为项目背景依据；如后续需要更完整地吸收调研结论，应单独做 OCR 摘要或人工摘录。

## 3. MVP 产品边界

第一版只验收 P0-M 闭环：让用户能更快找到可用港湾，并能反馈问题。

必须闭环：

- 自然语言或快捷按钮发起需求。
- 获取定位，或在拒绝定位时手动选择位置。
- 查询附近港湾。
- 按开放状态、设施类型、设施状态筛选。
- 按规则返回 1-3 个推荐结果，并解释推荐原因。
- 查看港湾详情。
- 发起导航，地图失败时降级展示地址或直线距离。
- 提交文字反馈。
- 自动生成工单。
- 记录对话日志和推荐日志，便于验收和复盘。

演示增强，但不阻断 MVP：

- 天气联动推荐。
- 地图可视化。
- 图片反馈上传。
- 简易管理页面。

暂不做：

- 复杂推荐模型训练。
- IoT、人流实时统计、室内实时温度。
- 完整图书借阅。
- 完整政府系统打通。
- 自动派单、智能补货。
- 强制登录、复杂权限、完整数据看板。

## 4. 推荐技术路线

由于当前仓库还没有代码，建议先采用“本地可运行 Web 原型 + 清晰后端边界”的方式开发。这样最适合作品集展示，也方便后续迁移到小程序、Coze 工作流或真实后端。

建议第一阶段技术栈：

- 前端：React + TypeScript + Vite。
- 样式：CSS Modules 或 Tailwind 二选一，优先保持轻量。
- 本地数据：JSON seed 数据起步，随后迁移 SQLite。
- 后端接口：先用前端内置服务层模拟，V0.5 再抽成 API。
- Agent 能力：第一阶段用规则意图识别实现可控演示，后续再接 Coze 或大模型 API。
- 地图与天气：第一阶段使用 mock 和降级逻辑，V0.5 接真实服务商。

选择理由：

- 可以最快跑通“找港湾 - 看详情 - 导航 - 反馈 - 工单”的闭环。
- 不依赖外部 API 就能先验收 P0-M 逻辑。
- 后续接 Coze、地图、天气、数据库时替换边界清楚。
- 适合 GitHub 展示和面试演示。

## 5. 版本路线

### V0.1：本地概念验证

目标：验证 Agent 是否能理解常见需求，并返回结构化推荐。

交付：

- 应用骨架。
- 样例港湾数据 10-30 条。
- 规则意图识别：喝水、充电、休息、避雨、如厕、反馈。
- 推荐规则：开放状态、设施匹配、设施状态、距离或模拟步行时间。
- 推荐卡片：名称、距离/步行时间、开放状态、设施、推荐理由。

验收：

- 输入“我想喝水”，能返回饮水设施正常且开放的港湾。
- 输入“下雨了还想充电”，能识别充电和室内条件。
- 输入“找个地方”，能追问一次并展示快捷选项。

### V0.5：可交互原型

目标：跑通用户查找港湾的主要页面链路。

交付：

- 首页。
- Agent 对话页。
- 港湾列表/地图降级页。
- 港湾详情页。
- 手动位置选择。
- 导航按钮与地图失败降级。
- 日志结构设计。

验收：

- 用户可以从首页或对话页完成一次推荐。
- 拒绝定位后仍可通过手动位置继续查询。
- 地图不可用时不阻断推荐。

### V1.0：MVP 闭环

目标：完成可演示的核心产品闭环。

交付：

- 文字反馈页面。
- 反馈记录生成。
- 工单生成。
- 简易管理入口：查看港湾、设施状态、反馈和工单。
- 天气联动演示能力。
- 图片反馈降级策略。
- P0-M 验收用例。

验收：

- 用户能完成“找港湾 - 看详情 - 导航 - 反馈”。
- 有效反馈能生成工单。
- 所有 P0-M 验收用例通过。

## 6. 功能工作包拆分

### A. 项目基础

- 初始化前端项目。
- 增加 README。
- 增加 `.gitignore`。
- 约定目录结构。
- 增加基础脚本：启动、构建、检查。

建议目录：

```text
src/
  app/
  components/
  data/
  features/
    agent/
    harbor/
    recommendation/
    feedback/
    work-order/
  lib/
  types/
  tests/
docs/
```

### B. 数据模型

优先建立 P0-M 必需数据：

- `harbor`：港湾基础信息、经纬度、开放时间、状态、更新时间。
- `facility`：设施类型、可用状态、故障状态、所属港湾。
- `report`：反馈内容、分类、图片可选、状态。
- `work_order`：工单编号、关联反馈、处理状态。
- `conversation_log`：Agent 识别记录。
- `recommendation_log`：推荐结果、排序因子、降级情况。

V0.1 可以先用 TypeScript 类型和 JSON 样例数据，V0.5 再迁移 SQLite 或后端数据库。

### C. Agent 规则层

先实现可控规则：

- 意图识别：找港湾、找设施、导航、反馈、模糊需求、安全/医疗边界。
- 参数提取：设施类型、天气场景、位置状态、是否需要室内、是否开放。
- 追问策略：缺位置或缺需求时最多追问一次。
- 安全边界：头晕、中暑等只推荐有应急药箱的港湾，并提示及时就医，不做医疗诊断。

后续接 Coze 或大模型时，规则层仍保留作为兜底和验收基线。

### D. 推荐引擎

实现可解释规则排序：

- 过滤关闭港湾。
- 过滤目标设施不存在的港湾。
- 排除设施故障或停用。
- 优先步行时间短。
- 设施匹配度更高优先。
- 空闲优先于适中，适中优先于较拥挤。
- 返回推荐原因和降级说明。

第一版不让大模型计算精确距离、步行时间、库存或实时状态。

### E. 用户端页面

优先页面：

- 首页：快捷需求、Agent 输入、附近港湾入口。
- 对话页：输入、追问、推荐卡片。
- 港湾详情页：开放状态、设施状态、更新时间、导航、反馈入口。
- 反馈页：分类、文字描述、图片可选但不阻断。

地图页可先做列表视图，后续增强地图可视化。

### F. 反馈与工单

先实现最小闭环：

- 文字反馈必可提交。
- 图片上传失败时允许继续提交文字反馈。
- 提交反馈后生成 `report_id`。
- 同步生成 `work_order_id`。
- 工单初始状态为 `pending`。

P1 再做反馈进度、通知和复杂处理流。

### G. 管理能力

MVP 只需要简易管理入口：

- 查看港湾数据。
- 修改港湾开放/临时关闭状态。
- 修改设施状态。
- 查看反馈列表。
- 修改工单状态。

完整后台、角色权限、看板、导出能力放入 P1。

### H. 测试与验收

优先覆盖 PRD 中 P0-M 验收标准：

- 喝水推荐。
- 多需求推荐。
- 模糊需求追问。
- 定位拒绝降级。
- 设施筛选。
- 开放状态判断。
- 推荐排序。
- 地图失败降级。
- 港湾详情。
- 文字反馈。
- 图片反馈降级。
- 工单生成。
- 数据更新时间提示。

## 7. 首轮开发任务清单

第一轮建议按以下顺序执行：

1. 建立项目骨架：React + TypeScript + Vite。
2. 新增 README、`.gitignore`、基础目录。
3. 从 PRD 抽取 P0-M TypeScript 类型。
4. 编写 4 条验收样例港湾数据：HB001、HB002、HB003、HB004。
5. 实现规则意图识别。
6. 实现推荐筛选和排序。
7. 做首页和推荐结果页面。
8. 做港湾详情页。
9. 做文字反馈和工单生成。
10. 增加 P0-M 单元测试。
11. 本地运行和截图验证。
12. 提交并推送到 GitHub。

## 8. 第一轮待确认事项

开始写代码前，建议确认以下事项：

- 首版展示形态：先做 Web/H5，还是直接做小程序？
- 地图服务商：高德、腾讯、百度，还是先 mock？
- 天气服务商：是否已有 API Key，还是先 mock？
- 是否优先接 Coze，还是先用规则 Agent 打样？
- 试点区域和真实港湾数据是否已有 Excel/表格？
- GitHub 仓库是否要把调研 PDF 一起纳入版本管理？当前 PDF 约 16 MB，建议先不纳入主仓库。

## 9. 推荐下一步

下一步建议进入 V0.1 开发：先搭建前端项目骨架，并实现“规则 Agent + 样例港湾推荐”。这是最小、最快、风险最低的第一块积木。

完成 V0.1 后，项目就能从文档状态进入可运行状态，后续再逐步接入数据库、地图、天气和 Coze。
