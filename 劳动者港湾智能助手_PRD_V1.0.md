## 1. 项目概述

### 1.1 项目简介

劳动者港湾智能助手面向户外劳动者和港湾运营管理方，提供“自然语言表达需求 - 智能识别服务条件 - 匹配附近开放港湾 - 导航前往 - 反馈问题 - 生成工单”的闭环服务。产品以 Coze Agent 作为自然语言理解和流程编排入口，以地图、天气、港湾数据库和管理后台作为能力支撑。

### 1.2 一句话产品定义

一个能听懂“我想喝水、充电、避雨、休息”等真实表达，并推荐附近可用劳动者港湾的公共服务智能助手。

### 1.3 项目背景

重庆市劳动者港湾数量持续增加，但从供需视角看，仍存在空间分布不均、服务设施差异大、信息入口分散、开放状态和设施状态不透明、问题反馈链路不畅等问题。户外劳动者往往在高温、降雨、低温、疲劳、手机低电量等场景下临时产生服务需求，需要一个低门槛、可解释、可导航、可反馈的统一入口。

### 1.4 需求来源

- 竞赛调研报告：《“港湾蒙尘，工驻何方？” - 供需视角下重庆市劳动者港湾建设及使用状况调研》。
- 产品提示词中整理的 10 类核心问题。
- 户外劳动者典型场景：喝水、充电、热饭、休息、避雨、如厕、应急药品、图书借阅、信息咨询。
- 运营管理场景：港湾信息维护、设施状态更新、物资库存、反馈工单处理、运营数据看板。

> 资料说明：当前 PDF 为纯图版，本文引用调研结论时不编造页码和具体数值；涉及页码、样本量、港湾数量、区县覆盖等定量信息统一标记为“待核对原始调研报告”。

### 1.5 核心问题

| 问题类别 | 具体表现 | 产品机会 |
|---|---|---|
| 供给分布 | 港湾数量增加，但区域分布和实际需求不完全匹配 | 用地图和数据沉淀形成空间化服务入口 |
| 信息认知 | 劳动者不知道附近港湾位置、开放时间和服务内容 | 用 Agent 和快捷入口降低查找门槛 |
| 入口分散 | 地图、公众号、小程序、App 信息不同步 | 建立统一港湾数据库和标准字段 |
| 设施差异 | 饮水、充电、空调、微波炉、药箱、洗手间等配置不一 | 用设施条件筛选和状态展示提升匹配准确性 |
| 状态不透明 | 到达前无法确认开放、设施正常、是否拥挤 | 展示开放状态、设施状态、更新时间和拥挤等级 |
| 天气影响 | 高温、降雨、低温会改变服务优先级 | 接入天气 API，自动补充需求条件 |
| 反馈断点 | 设施损坏、物资不足、责任不清、反馈渠道不畅 | 生成反馈工单，形成管理闭环 |

### 1.6 产品价值

- 对劳动者：减少寻找服务点的时间，提高到达后可用概率。
- 对管理方：统一港湾数据、反馈工单和运营状态，提升维护效率。
- 对公共服务：将“建设港湾”延伸为“可发现、可使用、可反馈、可优化”的数字化服务。
- 对作品集：展示用户研究、AI Agent 设计、Coze 工作流、地图天气 API、数据模型、MVP 规划和异常降级能力。

### 1.7 当前阶段

当前处于 MVP 产品方案设计阶段。尚未接入真实港湾数据、真实政府或工会系统、真实传感器和付费第三方接口。

### 1.8 产品愿景

从“附近有没有港湾”升级为“在当下环境和个人需求下，哪个港湾最适合我去”，并将用户反馈反向驱动港湾维护、服务补给和选址优化。

## 2. 调研结论与需求依据

### 2.1 港湾供给问题

| 结论 | 说明 | 来源 |
|---|---|---|
| 港湾数量增长但分布不均 | 不同区县、商圈、道路、劳动者聚集点周边存在服务密度差异 | 来源：竞赛调研报告，页码待核对原始调研报告 |
| 港湾服务半径与劳动者路线存在错配 | 外卖、快递、环卫、网约车等群体行动路径不同，固定点位不一定覆盖真实停留点 | 来源：竞赛调研报告，页码待核对原始调研报告 |
| 设施配置标准不完全一致 | 部分港湾具备饮水、充电、空调、微波炉、药箱等设施，部分仅提供基础休息 | 来源：竞赛调研报告，页码待核对原始调研报告 |

### 2.2 劳动者需求问题

劳动者的核心需求不是抽象地“找港湾”，而是在具体状态下获得可用服务：

- 天气炎热时：室内、空调、饮水、休息。
- 下雨时：避雨、距离近、可充电。
- 手机低电量时：充电插座或充电设备。
- 饭菜变凉时：微波炉或热饭设备。
- 身体不适时：应急药箱，并提示必要时及时就医。
- 工作间隙短时：步行时间短、开放状态确定、设施状态正常。

### 2.3 供需错配问题

| 错配类型 | 表现 | MVP 应对 |
|---|---|---|
| 空间错配 | 港湾离当前位置远，或步行路线不便 | 调用地图 API 计算实际步行时间 |
| 时间错配 | 用户需要时港湾关闭或临时停用 | 根据开放时间和临时关闭状态筛选 |
| 设施错配 | 港湾存在但没有目标设施 | 设施条件筛选 |
| 状态错配 | 有设施但故障、缺货或不可用 | 展示设施状态和更新时间 |
| 认知错配 | 用户不知道入口、不会查 | 自然语言 Agent 和快捷需求按钮 |

### 2.4 宣传和认知问题

现有信息入口可能分散在地图标注、公众号、小程序、App、线下标识等渠道，更新节奏不一致。MVP 需要建立统一数据源和统一展示规则，至少保证用户端看到“位置、开放时间、设施、状态、更新时间、导航入口”。

### 2.5 运营管理问题

港湾建设、管理和运营主体可能包括工会、城管、银行、企业、社区等，责任边界复杂。MVP 需要通过后台字段记录管理主体、联系电话、数据来源、更新时间和工单责任人。

### 2.6 反馈沟通问题

用户发现饮水机没水、插座故障、药箱缺药、卫生问题等情况时，需要低门槛反馈。系统应自动记录港湾、问题类别、图片、描述、上报时间、处理状态，并生成工单供管理人员处理。

### 2.7 天气对需求的影响

天气不是单独信息，而是需求条件的一部分。高温会提高“空调、饮水、室内休息”的优先级；降雨会提高“室内、距离近、避雨、充电”的优先级；低温会提高“室内、热水、休息”的优先级。

## 3. 产品目标与非目标

### 3.1 用户目标

| 目标 | 衡量方式 | 目标值 |
|---|---|---|
| 快速找到附近可用港湾 | 从输入需求到看到推荐结果的时间 | MVP 建议目标值：10 秒内，待验证 |
| 提高到达后设施可用概率 | 推荐结果中设施状态正常的比例 | MVP 建议目标值：80% 以上，待验证 |
| 降低复杂操作 | 完成一次推荐所需手动输入项 | MVP 建议目标值：不超过 2 项，待验证 |
| 便捷反馈问题 | 从详情页提交反馈的步骤数 | MVP 建议目标值：4 步内，待验证 |

### 3.2 业务目标

| 目标 | 说明 |
|---|---|
| 建立统一港湾数据底座 | 统一港湾位置、设施、开放、状态、管理主体字段 |
| 形成问题反馈闭环 | 用户反馈进入工单，管理后台可处理和回传结果 |
| 支撑港湾服务优化 | 沉淀需求、点击、导航、反馈、工单等数据 |
| 适合作品集展示 | 呈现 AI 产品经理从调研到方案、流程、数据和指标的完整能力 |

### 3.3 产品目标

- 支持自然语言需求识别和多意图识别。
- 支持用户定位、附近港湾查询、设施筛选、开放状态判断。
- 支持天气场景联动推荐。
- 支持地图展示、步行路线规划和外部导航。
- 支持文字和图片问题反馈，并生成工单。
- 支持管理人员维护港湾、设施、物资和工单状态。

### 3.3.1 第二轮审查后的 MVP 收敛原则

MVP 不追求一次性做成完整公共服务平台，而是优先证明“用户能不能更快找到可用港湾”。因此第一版按“P0-M 必须闭环、P0-E 演示增强、P1 运营增强、P2 远期规划”拆分。

| 层级 | 定义 | 是否阻断 MVP 验收 | 说明 |
|---|---|---|---|
| P0-M：MVP 必须闭环 | 没有该能力就无法完成“找港湾 - 看详情 - 去导航 - 提反馈” | 是 | 仅保留自然语言/快捷入口、定位或手动位置、港湾筛选推荐、详情、导航、文字反馈、工单生成 |
| P0-E：MVP 演示增强 | 能提升作品集完整度，但失败时不阻断核心链路 | 否 | 天气联动、图片上传、地图可视化可作为增强；接口失败时必须能降级 |
| P1：重要增强 | 有助于真实运营，但不影响首版验证 | 否 | 管理后台完整版、反馈进度、收藏、评价、库存精细管理、消息通知 |
| P2：远期规划 | 依赖硬件、组织协同或长期数据 | 否 | IoT、人流实时统计、自动派单、智能补货、选址分析 |

> 第二轮产品决策：首版验收只看 P0-M 是否跑通；P0-E 做到“可演示、可降级”即可，不作为上线阻断项。

### 3.4 MVP 验证目标

- 验证劳动者是否愿意通过自然语言或快捷按钮查找港湾。
- 验证规则推荐能否满足“近、开、可用、设施匹配”的核心诉求。
- 验证反馈工单能否提高管理方问题处理效率。
- 验证天气联动是否能提升推荐相关性。
- 验证零基础开发者能否用 Coze + 数据库 + API 完成可演示闭环。

### 3.5 非目标

- 不在第一版训练复杂推荐模型。
- 不在第一版接入实时人流传感器作为必需功能。
- 不在第一版实现完整图书借阅系统。
- 不在第一版实现全市级政府系统打通。
- 不让大模型自行计算精确距离、步行时间、库存或实时状态。
- 不把医疗咨询作为 Agent 能力范围。

### 3.6 第一版明确不解决的问题

| 问题 | 处理方式 |
|---|---|
| 实时人流精确统计 | 用“空闲、适中、较拥挤”三档替代 |
| 室内实时温度 | 管理员手动更新或显示暂无实时数据 |
| 药品实时库存 | 管理员手动盘点 |
| IoT 设备状态 | P2 阶段接入 |
| 自动派单 | P2 阶段探索 |
| 港湾选址优化 | 作为 V2.0 数据应用方向 |

### 4.4 Jobs To Be Done

- 当我在户外工作口渴时，我想快速找到附近有饮水设备且正在开放的港湾，以便少走冤枉路。
- 当我遇到下雨且手机快没电时，我想找到能避雨和充电的地方，以便继续完成工作。
- 当我发现港湾设施坏了时，我想用图片和文字快速反馈，以便管理人员及时处理。
- 当我是管理员时，我想看到待处理工单和港湾设施状态，以便维护服务质量。

### 4.5 高频场景

| 场景 | 用户表达 | 系统理解 |
|---|---|---|
| 喝水 | 我想喝水 | 饮水机、饮用水、正在开放、距离近 |
| 充电 | 手机没电了 | 充电插座、设施正常 |
| 热饭 | 饭凉了 | 微波炉、热饭设备 |
| 高温休息 | 天气太热了，想休息 | 高温、空调、室内、饮水、开放 |
| 下雨避雨 | 下雨了，附近哪里能避雨 | 降雨、室内、距离近、开放 |
| 问题反馈 | 饮水机没水了 | 反馈类别：物资不足或设施异常 |

### 4.6 低频但高风险场景

- 用户输入“头晕、中暑、不舒服”：系统可推荐有应急药箱的港湾，但必须提示必要时及时拨打急救电话或就医。
- 天气极端恶劣：系统应优先提示安全，不承诺港湾一定可用。
- 地图 API 或天气 API 失败：系统应说明当前信息不完整，提供可用的基础列表或手动选择区域。

### 4.7 用户旅程地图

| 阶段 | 用户动作 | 用户问题 | 系统能力 | 关键体验点 |
|---|---|---|---|---|
| 产生需求 | 口渴、疲劳、低电量、下雨 | 哪里能解决当下问题 | 快捷入口、Agent 输入 | 不要求用户知道“港湾”概念 |
| 寻找信息 | 打开小程序或 Agent | 附近有没有 | 定位授权、区域选择 | 定位失败可降级 |
| 查看推荐 | 看 1-3 个推荐 | 哪个最适合 | 推荐排序、理由解释 | 推荐理由可理解 |
| 选择港湾 | 点卡片或地图标记 | 是否真的开放可用 | 详情页、状态、更新时间 | 明确可信度 |
| 导航到达 | 调起路线 | 怎么走 | 地图 API、外部导航 | 低操作成本 |
| 使用设施 | 饮水、充电、休息 | 设施是否正常 | 设施状态展示 | 与现场一致性 |
| 反馈问题 | 上传图片和描述 | 怎么反馈 | 问题分类、工单生成 | 提交流程短 |
| 查看结果 | 查看处理进度 | 有没有处理 | 我的反馈、工单状态 | 状态透明 |

## 5. 产品范围

### 5.1 功能优先级

| 范围 | 功能 | 优先级 | 判断依据 |
|---|---|---|---|
| Agent | 自然语言需求识别、意图识别、参数提取、必要追问、结果解释 | P0-M：MVP 必须闭环 | 决定产品核心差异 |

| 港湾数据 | 基础信息、开放时间、设施配置、设施状态、更新时间、数据新鲜度 | P0-M：MVP 必须闭环 | 推荐准确性的基础 |
| 推荐 | 设施筛选、开放状态、规则排序、推荐卡片 | P0-M：MVP 必须闭环 | 完成服务匹配闭环 |
| 导航 | 步行时间、路线规划、外部导航；地图失败时展示地址和直线距离 | P0-M：MVP 必须闭环 | 用户需要实际前往 |
| 反馈 | 文字反馈、工单生成 | P0-M：MVP 必须闭环 | 建立最小问题闭环 |
| 地图可视化 | 地图标记、地图与列表联动 | P0-E：演示增强 | 列表可降级承载推荐，地图不是唯一入口 |
| 天气 | 实时天气查询、天气场景提醒、天气联动推荐 | P0-E：演示增强 | 强化场景化推荐，但 API 失败不阻断推荐 |

| 用户记录 | 收藏、历史记录、反馈进度 | P1：重要增强 | 提升复访和可信度 |
| 管理后台 | 港湾维护、设施状态、库存、工单处理、看板 | P1：重要增强 | 真实运营需要；MVP 可先用数据库表或低代码表格模拟 |
| 评价 | 港湾评价、满意度 | P1：重要增强 | 补充推荐因子 |
| 实时 IoT | 室内温度、人流、设备状态、智能补货、自动派单 | P2：远期规划 | 依赖硬件和预算 |
| 选址分析 | 需求热力图、选址辅助 | P2：远期规划 | 需要长期数据沉淀 |

### 5.2 暂不实现功能

| 功能 | 暂不实现原因 | 替代方案 |
|---|---|---|
| 精确实时人数 | 依赖传感器，部署成本高 | 管理员维护三档拥挤程度 |

## 7. 核心业务流程

### 7.1 自然语言需求识别流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 用户在 Agent 输入框输入自然语言，或点击快捷需求后补充文字 |
| 输入 | 用户文本、上下文、当前位置、时间、天气缓存 |
| 处理步骤 | 开始 -> 接收用户输入 -> 识别意图 -> 抽取实体参数 -> 判断是否缺少位置或关键条件 -> 必要时追问 -> 输出结构化需求 |
| 输出 | intent、facility_types、weather_context、distance_preference、open_now、location |
| 异常情况 | 无法识别、文本过短、包含医疗或危险内容 |
| 降级方案 | 提供快捷需求按钮；医疗问题提示及时就医；无法识别时追问“你想找喝水、充电、休息还是避雨？” |

### 7.2 最近港湾推荐流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 用户表达找港湾或具体设施需求 |
| 输入 | 结构化需求、用户位置、港湾数据库 |
| 处理步骤 | 开始 -> 获取用户位置 -> 查询候选港湾 -> 筛选正在开放 -> 筛选目标设施 -> 排除故障/停用/库存不足 -> 初步距离筛选 -> 调用地图 API -> 计算步行时间 -> 规则排序 -> 返回 1-3 个结果 |
| 输出 | 推荐卡片、推荐理由、步行时间、导航入口 |
| 异常情况 | 无定位、无候选、地图 API 失败 |
| 降级方案 | 手动选择区县或商圈；扩大搜索范围；使用直线距离并明确提示精度降低 |

### 7.2.1 MVP 推荐权重假设

第一版使用可解释规则推荐，不训练复杂推荐模型。排序权重为 MVP 阶段产品假设，后续需要通过真实使用数据、到达确认、用户反馈和现场核验持续调整。

| 排序因子 | 权重 | 说明 |
|---|---:|---|
| 步行时间 | 40% | 由地图 API 计算实际步行时间，不由大模型估算 |
| 设施匹配度 | 30% | 目标设施越完整、状态越正常，得分越高 |
| 开放状态与设施状态 | 15% | 正在开放、未临时关闭、设施未故障优先 |
| 拥挤程度 | 10% | 空闲优先于适中，适中优先于较拥挤；数据不足时不强扣分 |
| 用户评价 | 5% | MVP 可作为弱因子；评价数据不足时降低权重影响 |

### 7.3 天气与港湾联动推荐流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 用户表达“太热、下雨、冷”等天气场景，或首页天气触发提醒 |
| 输入 | 用户位置、天气 API 结果、用户文本 |
| 处理步骤 | 开始 -> 获取位置 -> 查询天气 -> 判断天气场景 -> 补充设施需求 -> 调用港湾推荐流程 -> 生成天气提醒 -> 输出推荐 |
| 输出 | 天气场景说明、推荐港湾、注意事项 |
| 异常情况 | 天气 API 失败 |
| 降级方案 | 只基于用户文本识别天气需求，不展示实时天气数据 |

### 7.4 港湾导航流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 用户点击推荐卡片或详情页的导航按钮 |
| 输入 | 用户位置、港湾经纬度、地图服务商 |
| 处理步骤 | 开始 -> 检查定位授权 -> 调用路线规划 -> 展示步行时间和路线 -> 用户确认 -> 调起外部导航 |
| 输出 | 路线、预计步行时间、外部导航入口 |
| 异常情况 | 定位失败、地图 API 失败、坐标缺失 |
| 降级方案 | 展示港湾地址，支持复制地址或打开地图搜索 |

### 7.5 图片问题上报流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 用户在详情页点击“反馈问题” |
| 输入 | 港湾 ID、问题分类、图片、文字描述、用户联系方式可选 |
| 处理步骤 | 开始 -> 选择港湾 -> 上传图片 -> 填写描述 -> 问题分类 -> 内容安全审核 -> 写入反馈表 -> 生成工单编号 -> 返回提交结果 |
| 输出 | report_id、work_order_id、提交成功提示 |
| 异常情况 | 图片上传失败、图片违规、重复提交 |
| 降级方案 | 允许仅文字提交；违规图片不保存并提示重新上传；重复提交合并或提示已有工单 |

### 7.6 工单处理流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 新反馈进入工单表 |
| 输入 | 工单信息、港湾管理主体、问题分类 |
| 处理步骤 | 开始 -> 管理员查看工单 -> 确认问题 -> 分配责任人 -> 更新处理状态 -> 填写处理结果 -> 通知用户 -> 完成工单 |
| 输出 | 工单状态、处理记录、用户通知 |
| 异常情况 | 无责任人、超时未处理、问题不属实 |
| 降级方案 | 进入待分配队列；超时提醒上级管理员；标记无效并说明原因 |

### 7.7 管理员更新设施状态流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 管理员巡检、收到反馈、物资盘点 |
| 输入 | 港湾 ID、设施类型、设施状态、库存、备注 |
| 处理步骤 | 开始 -> 登录后台 -> 选择港湾 -> 更新设施状态 -> 写入状态日志 -> 更新前端展示 -> 必要时关联工单 |
| 输出 | 设施状态、更新时间、更新人 |
| 异常情况 | 权限不足、字段缺失 |
| 降级方案 | 禁止提交并提示必填项；记录失败日志 |

### 7.8 用户拒绝定位时的降级流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 用户拒绝定位授权 |
| 输入 | 用户选择的区县、商圈、地址或文字位置 |
| 处理步骤 | 开始 -> 提示定位用途 -> 用户拒绝 -> 展示手动位置入口 -> 用户选择区域或输入地址 -> 地理编码 -> 查询港湾 |
| 输出 | 基于手动位置的推荐结果 |
| 异常情况 | 地址无法解析 |
| 降级方案 | 展示区县热门港湾列表，并提示步行时间不可用 |

### 7.9 无符合条件港湾时的处理流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 筛选后无结果 |
| 输入 | 用户需求、当前位置、筛选条件 |
| 处理步骤 | 开始 -> 判断无结果原因 -> 放宽非关键条件 -> 扩大距离 -> 返回替代方案 |
| 输出 | 无结果说明、替代港湾、修改条件入口 |
| 异常情况 | 数据库为空 |
| 降级方案 | 展示最近港湾并标注“不完全满足条件”；提示用户谨慎参考 |

### 7.10 第三方 API 失败时的降级流程

| 项目 | 内容 |
|---|---|
| 触发条件 | 地图、天气、对象存储、通知接口超时或失败 |
| 输入 | API 请求参数、错误码 |
| 处理步骤 | 开始 -> 记录失败日志 -> 自动重试 -> 判断是否可用缓存 -> 输出降级结果 |
| 输出 | 降级提示、缓存数据或基础数据 |
| 异常情况 | 连续失败 |
| 降级方案 | 地图失败用直线距离；天气失败不展示实时天气；图片失败允许稍后重试；通知失败保留站内状态 |

## 8. 功能需求详述

### 8.1 自然语言需求 - 设施条件映射规则

| 用户自然表达 | 显性需求 | 隐性/补充条件 | 对应设施或字段 | 默认筛选条件 |
|---|---|---|---|---|
| 我想喝水 | 饮水 | 距离近、正在开放 | 饮水机、饮用水 | open_now=true，facility_status=normal |
| 想充电 | 充电 | 可停留、室内优先 | 充电插座、充电设备 | open_now=true |
| 饭凉了 | 热饭 | 可进入室内 | 微波炉、饭菜加热 | facility_status=normal |
| 想休息 | 休息 | 桌椅、安静、拥挤程度低 | 桌椅、休息区、crowd_level | open_now=true |
| 太热了 | 避暑休息 | 空调、饮水、室内空间 | 空调、饮水机、室内空间 | weather_type=高温，open_now=true |
| 下雨了 | 避雨 | 室内、距离近 | 室内空间、雨具库存 | weather_type=降雨，按步行时间优先 |
| 想上厕所 | 如厕 | 开放且距离近 | 洗手间 | open_now=true |
| 头晕/中暑/不舒服 | 应急支持 | 提示及时就医 | 应急药箱、药品库存 | 不提供医疗诊断 |
| 手机没电 | 充电 | 可停留 | 充电设施 | facility_status=normal |
| 想找人少的地方 | 低拥挤 | 空闲优先 | crowd_level | 空闲 > 适中 > 较拥挤 |
| 哪里能借书 | 图书服务 | 港湾图书角 | 图书借阅、图书库存 | P1/P2 增强 |
| 有没有雨伞 | 雨具 | 库存可用 | 雨具库存 | stock_status=充足或未知提示 |
| 有没有热水 | 热水 | 饮用水温度 | 热水设备 | facility_status=normal |
| 附近有没有港湾 | 找港湾 | 距离近、开放 | harbor、opening_hours | open_now=true |

多条件输入示例：

| 用户输入 | 系统识别结果 |
|---|---|
| 下雨了，我还想找个地方给手机充电 | 天气场景=降雨；显性需求=充电；隐性需求=室内避雨；位置条件=附近；状态条件=正在开放；设施条件=充电插座、室内空间 |
| 太热了，想找个地方休息喝水 | 天气场景=高温；显性需求=休息、饮水；隐性需求=空调、室内；排序偏好=步行时间短且设施完整 |
| 饭凉了，附近哪里能热一下 | 显性需求=热饭；设施条件=微波炉；状态条件=正在开放且设施正常 |

## 9. Agent 能力设计

### 9.1 Agent 角色定位

Agent 是公共服务场景下的需求理解和服务匹配助手，负责听懂用户口语表达，转成结构化查询条件，调用工作流、数据库和外部 API，并用简洁、可解释的方式返回结果。Agent 不负责虚构港湾数据，不自行计算精确距离，不替代医疗建议。

### 9.2 系统提示词原则

- 以户外劳动者的真实表达为优先，不要求用户使用标准术语。
- 先解决当下需求，再解释推荐理由。
- 必要追问要短，只追问完成任务所需的最少信息。
- 推荐必须基于数据库、地图 API、天气 API 或明确的产品假设。
- 对无法确认的数据明确提示“暂无实时数据”或“待确认”。
- 对身体不适类表达，提供港湾应急设施信息，同时提示及时就医。
- 不输出未经接口或数据库确认的开放状态、库存、距离和步行时间。

### 9.3 Agent 能力边界

| 能力 | Agent 负责 | 不负责 |
|---|---|---|
| 语言理解 | 意图识别、参数提取、多轮追问 | 直接生成事实数据 |
| 推荐解释 | 说明为何推荐某港湾 | 自行计算精确路线 |
| 数据查询 | 调用工作流查询数据库 | 绕过数据库编造港湾 |
| 天气理解 | 将天气转为服务条件 | 替代天气 API |
| 反馈规范化 | 将用户描述归类和结构化 | 判断现场真实责任 |
| 安全防护 | 拒答危险内容、医疗提示 | 提供诊疗建议 |

### 9.6 上下文管理

- 同一轮会话保留最近一次位置、需求类型、候选港湾和推荐结果。
- 用户说“就去第一个”时，引用上一轮推荐列表第一个港湾。
- 用户说“换一个近一点的”时，保留设施条件，调整排序偏好。
- 用户连续反馈问题时，默认沿用当前详情页港湾，但提交前需让用户确认。

### 9.7 工作流路由

| 触发条件 | 路由 |
|---|---|
| 找服务点、找设施、天气下找港湾 | 工作流 A：港湾推荐 |
| 高温、降雨、低温相关表达 | 工作流 B：天气联动推荐 |
| 问题反馈、图片上传 | 工作流 C：问题反馈 |
| 管理员处理反馈 | 工作流 D：工单处理 |
| 只问港湾详情 | 港湾详情查询 |
| 问天气 | 天气查询 |
| 问医疗诊断 | 安全拒答 + 就医提示 |

### 9.8 知识库和数据库调用规则

- 港湾位置、开放时间、设施状态、库存、更新时间必须从数据库查询。
- 服务规则、使用说明、港湾政策说明可来自知识库。
- 当知识库与数据库冲突时，以数据库实时字段为准，并提示数据更新时间。
- 当数据库无值时，显示“暂无数据”或“待调研确认”，不由大模型补全。

### 9.9 API 调用规则

- 地图 API 负责定位、坐标、周边查询、步行路线、步行时间和导航。
- 天气 API 负责当前天气、温度、降雨、风力和预报。
- 对象存储负责图片保存和访问地址。
- 消息通知负责工单通知和处理结果提醒。
- API 失败时必须记录日志、重试并进入降级流程。

### 9.10 拒答和安全规则

- 不提供医疗诊断、处方或药品服用建议。
- 不承诺港湾一定开放或设施一定可用，只展示数据状态和更新时间。
- 不输出用户隐私、管理员联系方式以外的敏感信息。
- 不生成攻击、违法、歧视或骚扰内容。
- 图片内容需经过安全审核，违规图片不入库。

### 9.11 结果解释规则

推荐结果需包含：

- 港湾名称。
- 推荐理由：如“步行约 6 分钟，有饮水机，当前开放，信息 20 分钟前更新”。
- 关键状态：开放、设施正常、拥挤程度。
- 风险提示：如“拥挤程度为人工更新，可能与现场略有差异”。
- 下一步操作：查看详情、导航、换一批、反馈问题。

### 9.12 幻觉防控机制

- 所有港湾事实必须带 harbor_id。
- 推荐结果必须来自查询结果集合。
- 对数量、页码、价格、接口限额等未确认信息标注“待确认”。
- Agent 回复中不得出现未在数据库中的港湾名称。
- 对外部 API 返回空值时，不把空值解释为事实不存在。

## 10. Coze 工作流设计

### 10.1 工作流 A：港湾推荐

流程：开始 -> 获取用户输入 -> 意图识别 -> 参数提取 -> 判断是否有定位 -> 查询港湾数据库 -> 筛选开放状态 -> 筛选设施 -> 初步距离筛选 -> 调用地图 API -> 计算步行时间 -> 推荐排序 -> 生成推荐结果 -> 结束

| 项目 | 内容 |
|---|---|
| 输入参数 | user_input、user_location、need_type、facility_types、distance_preference、time |
| 输出参数 | recommendation_list、reason_list、fallback_message |
| 节点说明 | 意图识别节点输出结构化需求；数据库节点查询候选港湾；地图节点计算步行时间；规则节点排序 |
| 判断条件 | 是否有位置；是否有目标设施；是否开放；是否设施正常；是否有候选结果 |
| 数据来源 | harbor、facility、inventory、review、harbor_status_log、地图 API |
| 失败重试 | 地图 API 超时重试 1-2 次，待技术调研确认 |
| 降级策略 | 用直线距离初筛，提示步行时间暂不可用 |
| 日志记录 | 输入意图、筛选条件、候选数量、推荐结果、API 状态 |
| 验收标准 | 对“我想喝水”返回 1-3 个有饮水设施且开放的港湾 |

### 10.2 工作流 B：天气联动推荐

流程：开始 -> 获取位置 -> 查询天气 -> 判断天气场景 -> 补充设施需求 -> 调用港湾推荐工作流 -> 生成天气提醒 -> 输出推荐结果 -> 结束

| 项目 | 内容 |
|---|---|
| 输入参数 | user_location、user_input、weather_keyword |
| 输出参数 | weather_context、facility_types、recommendation_list、weather_tip |
| 节点说明 | 天气节点查询温度和降雨；规则节点把天气映射到设施条件 |
| 判断条件 | 高温、降雨、低温、强风等阈值待技术和业务确认 |
| 数据来源 | 天气 API、港湾推荐工作流 |
| 失败重试 | 天气 API 超时重试，失败后仅依据用户文本 |
| 降级策略 | 不展示实时天气，只按用户表达“下雨/太热”补充条件 |
| 日志记录 | 天气结果、补充条件、推荐结果 |
| 验收标准 | 输入“太热了想休息”时优先推荐室内、空调、饮水、开放港湾 |

### 10.3 工作流 C：问题反馈

流程：开始 -> 获取港湾 -> 上传图片 -> 获取文字描述 -> 问题分类 -> 生成规范描述 -> 写入反馈数据库 -> 生成工单编号 -> 通知管理人员 -> 返回结果 -> 结束

| 项目 | 内容 |
|---|---|
| 输入参数 | harbor_id、user_id、category、description、image_file、contact |
| 输出参数 | report_id、work_order_id、report_status |
| 节点说明 | 图片节点上传对象存储；分类节点归类问题；数据库节点写入 report 和 work_order |
| 判断条件 | 是否有港湾；图片是否合规；是否重复提交 |
| 数据来源 | report、work_order、对象存储、内容安全审核 |
| 失败重试 | 图片上传失败支持重试；数据库写入失败提示稍后再试 |
| 降级策略 | 允许仅文字反馈；通知失败不影响工单生成 |
| 日志记录 | 提交时间、港湾、分类、图片审核结果、工单号 |
| 验收标准 | 反馈“饮水机没水了”后生成工单并可在我的反馈查看 |

### 10.4 工作流 D：工单处理

流程：开始 -> 管理员查看工单 -> 确认问题 -> 分配责任人 -> 更新处理状态 -> 填写处理结果 -> 通知用户 -> 完成工单

| 项目 | 内容 |
|---|---|
| 输入参数 | work_order_id、admin_id、assignee_id、status、result |
| 输出参数 | updated_status、notification_result |
| 节点说明 | 后台列表展示工单；状态节点流转；通知节点回传处理结果 |
| 判断条件 | 管理员权限、工单状态、是否超时 |
| 数据来源 | work_order、report、user、harbor |
| 失败重试 | 通知失败重试，失败后保留站内状态 |
| 降级策略 | 无责任人时进入待分配；超时提醒上级管理员 |
| 日志记录 | 操作人、操作时间、状态变更、处理结果 |
| 验收标准 | 管理员可将工单从待处理更新为已完成，用户端状态同步 |

## 14. 非功能需求

| 类别 | 要求 |
|---|---|
| 性能 | 常规查询结果建议 3 秒内返回；涉及地图和天气 API 的完整推荐建议 10 秒内返回，待验证 |
| 可用性 | 核心查询、详情、反馈链路在弱网下应有加载和重试 |
| 稳定性 | 第三方 API 失败不应导致整个 Agent 不可用 |
| 数据准确性 | 精确距离、步行时间、天气、库存必须来自对应系统或明确标注暂无数据 |
| 信息时效性 | 港湾、设施、库存、拥挤程度需展示更新时间 |
| 可解释性 | 推荐结果必须说明主要排序原因 |
| 无障碍设计 | 字体、按钮、对比度适合强光、雨天、戴手套等环境 |
| 弱网环境 | 允许文本优先加载，图片延迟加载 |
| 低端手机适配 | 页面避免重动画，地图失败时可用列表 |
| 隐私保护 | 定位、手机号、图片按最小必要原则采集 |
| 图片安全审核 | 反馈图片需审核后入库或展示 |
| 定位权限 | 明确说明用途，拒绝后可手动选择位置 |
| 数据权限 | 管理员只能维护授权范围内港湾 |
| 操作日志 | 管理端关键操作必须记录 |
| 管理员权限分级 | 支持超级管理员、区域管理员、港湾管理员 |

> 产品决策：第一版不接入实时人数传感器，使用“空闲、适中、较拥挤”三档状态替代。

> 体验要求：户外劳动者可能数字化能力有限，且可能在强光、雨天、戴手套场景下使用；按钮和字体需要清晰，核心功能应支持快捷入口。

## 15. 异常场景与降级方案

| 异常场景 | 影响 | 降级方案 |
|---|---|---|
| 用户拒绝定位 | 无法计算附近和步行时间 | 提供区县、商圈、地址输入 |
| 定位失败 | 位置不准确或无位置 | 重新定位、手动位置、热门区域 |
| 天气 API 失败 | 无法展示实时天气 | 按用户文本识别天气需求，不阻断推荐 |
| 地图 API 失败 | 无法计算路线和步行时间 | 使用直线距离，提示精度降低 |
| 港湾数据为空 | 无法推荐 | 提示暂无数据，提供待确认事项 |
| 附近无符合条件港湾 | 用户无法获得完全匹配 | 扩大范围或返回最近但不完全满足的港湾 |
| 港湾已经关闭 | 推荐不可用 | 默认过滤，详情页标注关闭 |
| 设施临时故障 | 到达后不可用 | 过滤该设施，提示其他港湾 |
| 数据长期未更新 | 可信度下降 | 显示更新时间和谨慎参考提示 |
| 图片上传失败 | 反馈不完整 | 允许仅文字提交或稍后补图 |
| 图片内容违规 | 合规风险 | 拦截图片并提示重新上传 |
| 用户输入无法识别 | 对话失败 | 展示快捷需求选项 |
| 用户询问医疗问题 | 安全风险 | 不诊断，提示就医或急救 |
| 用户重复提交反馈 | 工单冗余 | 识别相同港湾同类问题，提示已有工单或合并 |
| 管理员长期未处理 | 闭环断裂 | 超时提醒、升级给区域管理员 |
| 用户网络较差 | 页面卡顿 | 文字优先、重试、缓存最近查看港湾 |

| AC-001 | 自然语言喝水推荐 | Given HB001、HB002、HB003、HB004 测试数据已入库且用户位置有效；When 用户输入“我想喝水”；Then conversation_log.intent=query_facility，extracted_params.facility_type=drinking_water，推荐结果包含 HB001，不包含 HB002 和 HB003，HB004 不计入满足条件结果。 |
| AC-002 | 多需求推荐 | Given 用户位置有效，天气 API 不可用；When 用户输入“下雨了，我还想找个地方给手机充电”；Then 系统识别 charging 和 indoor 条件，weather_api_status=failed 或 not_called 时仍返回可充电且室内的港湾。 |
| AC-003 | 模糊需求追问 | Given 用户未表达明确设施；When 用户输入“我想找个地方”；Then 系统只追问一次，并提供喝水、充电、休息、避雨、如厕快捷选项。 |
| AC-004 | 定位授权 | Given 用户首次使用；When 系统需要推荐附近港湾；Then 系统请求定位授权，并说明定位用于查找附近港湾；授权成功后写入推荐工作流输入。 |
| AC-005 | 定位拒绝降级 | Given 用户拒绝定位；When 用户继续查找港湾；Then 系统提供区县、商圈或地址输入，并在用户选择后继续推荐。 |
| AC-006 | 设施筛选 | Given HB001 有充电设施且 normal，HB002 无目标设施；When 用户查询充电；Then 只有设施 available=true 且 facility_status=normal 的港湾进入满足条件结果。 |
| AC-007 | 开放状态判断 | Given HB003 当前 closed 或 temp_closed；When 用户查询附近港湾；Then HB003 不进入推荐结果，详情页展示关闭状态和 status_reason。 |
| AC-008 | 推荐排序 | Given HB001 步行 6 分钟且设施正常，HB004 步行 8 分钟但目标设施故障；When 用户查询喝水；Then HB001 排名高于 HB004，recommendation_log.score_detail 记录排序因子。 |
| AC-009 | 地图路线 | Given 用户和港湾坐标有效且地图 API 正常；When 用户点击导航；Then 系统返回路线或调起外部导航，并记录 navigation_click。 |
| AC-010 | 地图失败降级 | Given 地图 API 超时；When 用户查询推荐；Then 系统可使用直线距离或地址降级，并提示“步行时间暂不可用”，recommendation_log.fallback_used=true。 |
| AC-011 | 港湾详情 | Given 用户点击 HB001 推荐卡；When 进入详情页；Then 展示 name、address、opening_hours、harbor_status、facility 列表、updated_at、last_verified_at 和反馈入口。 |
| AC-012 | 文字反馈 | Given 用户在 HB001 详情页；When 提交“饮水机没水”且不上传图片；Then report.image_url 允许为空，系统生成 report_id 和 work_order_id。 |
| AC-013 | 图片反馈降级 | Given 图片上传失败；When 用户选择继续提交文字反馈；Then 系统生成反馈和工单，不因 image_url 为空阻断 P0-M。 |
| AC-014 | 工单生成 | Given 有效反馈提交成功；When 系统写入 report；Then 自动生成 work_order，status=pending，report_id 与 work_order.report_id 一致。 |
| AC-015 | 数据更新时间 | Given HB001 data_freshness_status=stale；When 用户查看详情；Then 系统展示“信息更新时间较早，请谨慎参考”。 |
